# 코드 작성 원칙

## 같이 실행되지 않는 코드는 분리한다

동시에 실행되지 않는 코드가 하나의 함수나 컴포넌트에 섞이면 분기가 늘어나고 파악이 어려워진다. 조건에 따라 동작이 완전히 달라진다면 각각 별도의 함수나 컴포넌트로 분리한다.

## 구현 상세는 추상화한다

- 코드를 읽는 사람이 한 번에 고려할 수 있는 맥락은 제한되어 있다. 불필요한 구현 상세는 함수나 컴포넌트 뒤로 숨기고, 읽는 사람이 지금 이 문맥에서 꼭 알아야 할 것만 노출한다.
- 함께 자주 수정되는 코드는 가까이 둔다. 버튼과 클릭 핸들러처럼 응집도가 높은 코드가 멀리 떨어져 있으면, 한쪽을 수정할 때 다른 쪽을 놓치기 쉽다. 물론 핸들러 로직이 길어질 경우 가독성있게 분리한다.

## 로직 종류가 아닌 기능 단위로 쪼갠다

- 쿼리 파라미터, 상태, API 호출 같은 로직의 종류를 기준으로 함수나 Hook을 묶지 않는다. 그렇게 하면 한 함수가 담당하는 맥락이 무제한으로 늘어나고, 수정 영향 범위를 예측하기 어려워진다.
- 기능이나 도메인 단위로 분리하면 이름이 명확해지고, 수정 시 영향 범위가 좁아진다. 하나의 Hook이 페이지 전체 상태를 담당하면 새 파라미터가 생길 때마다 무조건 그 Hook을 건드리게 되고, 수정 영향이 어디까지 미치는지 추적하기 어려워진다.

## 복잡한 조건에는 이름을 붙인다

- 조건식이 복잡할수록 의도를 파악하기 어려워진다. 의미 있는 변수명으로 조건을 분리하면 코드를 따라가지 않아도 의도를 바로 읽을 수 있다.

- 다음 경우에는 이름을 붙이는 것이 좋다:
    - 조건 로직이 여러 줄에 걸쳐 복잡하게 얽혀 있을 때
    - 동일한 조건이 여러 곳에서 재사용될 때
    - 해당 조건만 독립적으로 테스트해야 할 때

단순하거나 한 번만 쓰이는 조건은 굳이 분리하지 않아도 된다.

## 매직 넘버에는 이름을 붙인다

- 의미 없이 숫자를 그대로 쓰면 맥락을 알 수 없어 잘못 수정될 위험이 있다. 숫자의 의도가 명확하지 않다면 상수로 선언해 이름을 붙인다.

```javascript
const MIN_AGE = 19
const MAX_AGE = 75
const DELAY_SECOND = 120
```

## 시점 이동을 줄인다

- 코드를 읽을 때 위아래를 왔다 갔다 하거나 여러 파일과 함수를 넘나들수록 맥락을 파악하는 데 시간이 걸린다. 가능하면 위에서 아래로, 한 함수 안에서 읽을 수 있도록 작성한다.
- 불필요한 추상화나 간접 참조가 시점 이동을 만드는 경우가 많다. 조건이 단순하다면 객체나 맵으로 감추기보다 조건을 그대로 드러내는 편이 읽기 쉽다.
- 대표적으로 컴포넌트 분기처리로 렌더링 시에 switch문으로 나타내는 방법이 있다.

```tsx
function StatusMessage({ status }: { status: "loading" | "error" | "success" }) {
  switch (status) {
    case "loading":
      return <Spinner />;
    case "error":
      return <ErrorMessage />;
    case "success":
      return <SuccessMessage />;
  }
}
```

## 삼항 연산자는 단순하게 쓴다

- 삼항 연산자가 중첩되면 조건 구조를 한눈에 파악하기 어려워진다. 조건이 복잡하다면 if문으로 풀고 early return을 사용해서 명확하게 드러낸다.

## 범위 조건은 왼쪽에서 오른쪽으로 읽히게 쓴다

- 범위를 조건은 부등식처럼 `min <= value && value <= max` 순서로 작성한다. `value >= min && value <= max`처럼 쓰면 중간값을 두 번 확인해야 해서 직관적이지 않다.

## 같은 이름은 같은 동작을 보장한다

- 같은 이름의 함수나 변수가 다른 동작을 하면 읽는 사람에게 혼란을 준다. 기존 라이브러리나 인터페이스와 이름이 겹치지 않도록 하고, 동작 차이가 있다면 이름에서 그 차이를 드러낸다.

결국 개발자가 예측가능한 코드를 작성하여 맥락을 잘 파악할 수 있도록 한다.

## 같은 종류의 함수는 반환 타입을 통일한다

- 같은 역할을 하는 함수나 Hook이 서로 다른 타입을 반환하면 일관성이 깨져 읽는 사람이 매번 반환 타입을 확인해야 한다. API Hook은 Query 객체를, 유효성 검사 함수는 `{ ok, reason }` 형태를 일관되게 반환하는 식으로 종류별 반환 타입을 맞춘다.

타입을 통일하면 TypeScript의 타입 좁히기를 활용할 수 있어 잘못된 접근을 컴파일 단계에서 막을 수 있다.

## 숨은 로직은 드러낸다

- 함수의 이름, 파라미터, 반환 타입으로 예측할 수 없는 동작을 함수 안에 숨기지 않는다. 숨은 로직은 예상치 못한 부수 효과를 만들고, 오류 발생 시 원인을 찾기 어렵게 한다.
- 함수는 이름에서 예측되는 동작만 수행하고, 로깅이나 추가 작업은 호출하는 쪽에서 명시적으로 처리한다.

## 함께 수정되는 파일은 같은 디렉토리에 둔다

- 파일을 모듈 종류(hooks, components, utils 등)로 나누지 말고, 함께 수정되는 파일을 도메인이나 기능 단위로 모은다. 의존 관계가 디렉토리 구조에 드러나면 잘못된 참조를 쉽게 발견할 수 있고, 기능을 삭제할 때 디렉토리 하나를 지우는 것으로 정리된다.

# SOLID 원칙

## SRP: 컴포넌트는 하나의 책임만 갖는다

컴포넌트나 Hook, 유틸 함수는 동작의 종류가 아닌 책임(도메인)을 기준으로 설계한다. 디자인, UX, 마케팅처럼 이해관계자가 다르다면 각자의 관심사를 별도 컴포넌트로 분리하는 것도 좋은 방법이다. 책임이 명확하지 않으면 코드의 의도가 불분명해지고 협업이 어려워진다.

## OCP: 변경보다 확장으로 대응한다

요구사항이 바뀔 때 기존 코드를 직접 수정하기보다 확장할 수 있는 구조를 만든다. 상태나 타입에 따라 분기가 늘어나는 경우라면, 매핑 테이블이나 설정 객체로 분리해 새 케이스를 추가만 하면 동작하도록 설계한다.

## LSP: 컴포넌트를 확장할 때 부모의 계약을 깨지 않는다

기존 컴포넌트를 감싸 새 컴포넌트를 만들 때는 부모의 props를 모두 수용해야 한다. 일부 props만 받으면 부모 컴포넌트로 교체할 수 없는 닫힌 컴포넌트가 된다. `ComponentProps`를 활용해 부모의 인터페이스를 상속하고, 추가 동작은 props를 오버라이드하는 방식으로 구현한다.

## ISP: 필요한 props만 의존하도록 인터페이스를 분리한다

컴포넌트가 사용하지 않는 props를 받도록 강제하지 않는다. 하나의 범용 인터페이스보다 역할에 맞게 쪼갠 인터페이스가 낫다. Compound Component 패턴처럼 조합 가능한 구조로 설계하면 각 사용처에서 필요한 부분만 가져다 쓸 수 있다.

## DIP: 컴포넌트는 구체적인 구현이 아닌 추상화에 의존한다

컴포넌트 안에서 API 호출이나 스토리지 접근 같은 구체적인 구현을 직접 다루지 않는다. Hook으로 추상화 계층을 두면 컴포넌트는 데이터를 어떻게 가져오는지 알 필요가 없고, 구현 세부사항이 바뀌어도 컴포넌트는 영향을 받지 않는다.
